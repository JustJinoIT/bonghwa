# 봉화(Bonghwa) — 기술 교훈

검증 방법론·CI 설계에서 얻은, 코드나 NOTES.md만 봐서는 안 드러나는 교훈.
"왜 이렇게 했는지"를 남겨서 같은 실수를 반복하지 않기 위한 기록.

## 2026-07-12 — 타이밍 기반 경합 테스트는 비결정적이다, 결정론적 스트레스 테스트로 대체하라

**발견**: push-트리거 배포와 수동 rollback의 동시 실행을 검증하려고 처음엔
`Start-Sleep -Milliseconds 300`으로 지연을 줘서 겹치길 "기대"했다. 실제로는
git credential-manager의 host-provider 자동탐지가 ~4초 지연을 만들어 매번
순차 실행이 됐고, 테스트는 통과했지만 아무것도 증명하지 못했다.

**원인**: 실제 운영 경로(git push)의 타이밍은 CI 환경마다, 실행마다 달라지는
외부 변수(네트워크, credential helper, 러너 부하)에 의존한다. 이걸로 "경합이
안전한지"를 검증하려는 것 자체가 잘못된 접근이었다.

**교훈**: 검증하려는 게 "공유 자원(같은 파일/junction)에 대한 동시 접근이
안전한가"라면, 실제 트리거 경로(git push)의 타이밍에 기대지 말고 **그 공유
자원에 접근하는 실제 코드(`rollback.ps1`)를 병렬 job으로 반복 실행해 경합을
강제로 만드는 결정론적 스트레스 테스트**를 짜야 한다. 이게 `VERIFY 7/7`이고,
실제로 16회 중 5회 경합을 재현해 `rollback.ps1`의 진짜 버그(mklink 실패 시
성공으로 오보고)를 잡아냈다. 타이밍 기반 테스트(`VERIFY 6/7`)는 "실제 운영
경로에서도 문제없다"는 별도의 증거로는 남겨두되, 버그를 잡는 주력은 항상
결정론적 스트레스 테스트 쪽이어야 한다.

## 2026-07-12 — 누적 상태(history.log)를 검증할 때는 구간 스냅샷을 떠라

**발견**: `KEEP_RELEASES` 정리 검증에서 "이번 구간에 `DEPLOY_FAIL`이 있으면
버그"라는 체크를 넣었는데, `history.log` 전체를 훑는 바람에 이전 스텝(VERIFY
3/6)이 의도적으로 만든 `DEPLOY_FAIL`까지 걸려서 오탐 실패했다.

**교훈**: 한 CI job 안에서 여러 검증 스텝이 같은 append-only 로그 파일을
공유할 때, 각 스텝은 자기 구간 시작 시점의 줄 수를 기억해두고 그 이후
추가분만 검사해야 한다. "이 파일에 X가 있으면 안 된다" 류의 체크는 전체
파일이 아니라 항상 "이번에 새로 추가된 부분"으로 스코프를 좁혀라.

## 2026-07-12 — branch protection은 CI 트리거 타이밍과 맞아야 의미가 있다

**발견**: "머지 전 CI 통과 필수"(`required_status_checks`)를 걸려다 말았다.

**원인**: 이 워크플로우는 `on: push`로, 커밋이 브랜치에 올라간 *이후에* 도는
구조다. GitHub의 required status checks는 PR 머지 시점에 "이미 통과한 체크가
있는지"를 확인하는 것이라, "push 이후에만 도는" CI와는 맞물리지 않는다 —
걸면 오히려 모든 직접 push가 막혀버린다(아직 아무 체크도 안 돌았으니까).

**교훈**: CI가 push-then-verify 구조(먼저 올리고 나중에 검증)인 저장소에
"머지 전 CI 필수"를 걸려면 워크플로우 트리거 자체를 PR 기반으로 바꿔야 한다.
그 전환 없이 branch protection만 강하게 걸면 셀프 락아웃이 된다. 지금은
force-push/브랜치 삭제만 막는 약한 보호로 유지 — 워크플로우 변경 없이 걸 수
있는 최대치.
